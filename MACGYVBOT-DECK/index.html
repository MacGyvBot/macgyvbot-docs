01
Command

한국어 명령을 안정적인 robot task로 바꾸는 과정

문제

자연어는 정해진 문법으로 들어오지 않는다

사용자는 “드라이버 가져다줘”, “드라이버 좀”, “이거 정리해”처럼 다르게 말한다.

초기 접근

LLM prompt로 구조화 command 생성

action, tool_name, target_mode로 정규화하면 task coordinator가 자연어를 몰라도 된다고 봤다.

한계

안전 명령까지 LLM에 맡기기 어렵다

pause/resume/exit는 latency와 오해석이 곧 safety risk가 된다.

방향 수정

fast parser와 LLM fallback 분리

안전 명령은 local path, 일반 명령은 hard parser + LLM fallback으로 처리했다.

결과

자연어 입력이 안정적인 task intent로 닫힘

deictic return은 current tool context를 함께 본다.

02
Tool Grasp Perception

탐지 bbox를 실제 grasp parameter로 바꾸는 과정

문제

탐지는 곧 파지가 아니다

YOLO bbox는 공구 위치를 주지만 잡을 점, yaw, RG2 width를 직접 주지 않는다.

초기 접근

bbox center를 pick target으로 사용

빠르고 단순하지만 공구의 긴 축과 실제 표면을 설명하지 못했다.

한계

서랍 벽, depth noise, yaw 불안정

벽 가까이 있거나 깊은 안쪽의 공구는 bbox center만으로는 잡기 어렵다.

방향 수정

SAM/Depth mask + PCA

mask pixels의 주축으로 yaw를 추정하고 depth/intrinsics로 width를 계산했다.

결과

perception output이 robot parameter가 됨

MoveIt target, J6 yaw, RG2 width가 pipeline으로 연결된다.

03
Handoff Perception

사용자가 실제로 공구를 잡았는지 판단하는 과정

문제

안전하게 전달하려면 무엇을 봐야 할까?

release 순간의 false positive는 로봇이 공구를 너무 일찍 놓는 문제다.

초기 접근

YOLO bbox + hand landmark rule

거리, ROI overlap, depth score가 threshold를 넘으면 grasp로 판단했다.

한계

bbox는 실제 공구가 아니고 proximity는 grasp가 아니다

open palm이나 스쳐 지나가는 손도 score를 넘을 수 있었다.

방향 수정

ML gesture + SAM/Depth contact

손 모양과 실제 공구 표면 접촉을 분리해서 검증했다.

결과

release 판단이 safety gate로 바뀜

rule score 단독 release를 제거하고 confidence/stability/contact를 함께 본다.

04
Drawer & Manipulation

공구 반납을 drawer ownership까지 닫는 과정

문제

사용한 공구를 어떻게 다시 제자리로?

가져다주는 것만으로는 공구 관리 루프가 닫히지 않는다.

초기 접근

서랍 내부를 vision으로 해석

열린 drawer area와 내부 배치를 인식해 놓을 위치를 계산하려 했다.

한계

서랍 전체 인식은 과한 문제 설정

반납에는 서랍 전체 이해보다 안정적인 기준점이 필요했다.

방향 수정

ArUco center + hand-eye + z clamp

returned tool bbox center를 marker center에 맞추는 문제로 축소했다.

결과

return flow가 drawer ownership까지 닫힘

receive → relabel → marker resolve → place → close로 정리된다.

05
Safety & Robustness

실패를 감지하고 안전하게 회복하는 과정

문제

실패가 발생해도 안전해야 한다

drop, planning fail, handoff timeout, pause/resume은 서로 다른 recovery가 필요하다.

초기 접근

단계별 retry/stop

각 perception/motion 단계에서 실패를 잡고 재시도하도록 처리했다.

한계

실패는 queue와 hardware state를 같이 흔든다

drawer open 상태, collision scene, 실제 gripper/tool state가 어긋날 수 있다.

방향 수정

TaskStep queue + runtime gate

pause는 queue 보존, resume은 suspended step 재실행, drop은 recovery queue로 연결한다.

결과

failure → detection → response로 설명 가능

실패를 숨기지 않고 status/log/GUI로 사용자에게 보여준다.
